# DRY Refactoring Plan

## Goals

1. Extract UI components to `@anaqio/ui` package
2. Create `@anaqio/schemas` package for shared Zod validations
3. Create error utility for consistent Supabase error handling
4. Dedup Supabase queries to use `@anaqio/supabase` consistently
5. Enhance `@anaqio/utils` with additional shared utilities

---

## Phase 3: Create Error Utility

### 3.1 Add to @anaqio/utils

```typescript
// packages/utils/src/error.ts
export function createDbError(table: string, operation: string) {
  return (error: { message: string }) => new Error(`[${table}.${operation}] ${error.message}`)
}

export function createApiError(endpoint: string) {
  return (error: string) => new Error(`[api.${endpoint}] ${error}`)
}
```

## Phase 4: Dedup Supabase Queries

### 4.1 Audit current Supabase usage

The `@anaqio/supabase` package already exists with:

- `createBrowserSupabaseClient(url, key)`
- `createServerSupabaseClient(url, key)`
- `createAdminSupabaseClient(url, key)`

### 4.2 Issues found

- Apps import from `@lib/supabase/*` local files, not `@anaqio/supabase`
- Some inline `supabase.from()` calls in actions

### 4.3 Actions

1. Ensure all apps use `@anaqio/supabase` for client creation
2. Ensure Server Actions get client via `createServerSupabaseClient`
3. Remove duplicate `lib/supabase/` directories in apps
